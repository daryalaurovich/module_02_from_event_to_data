Шаг 3. Добавление ограничений (Constraints)

Для сущностей sales и products определите следующие ограничения (которые необходимо будет реализовать при создании таблиц):

1. Укажите 3–4 атрибута, которые не могут быть пустыми (NOT NULL), и обоснуйте свой выбор.

Для sales атрибуты, которые не могут быть пустыми:
- sales_id (необходим для однозначной идентификации записи)
- product_id (определяет, какой товар был продан)
- quantity (показывает количество проданного товара)
- sales_timestamp (фиксирует момент совершения продажи)
Эти поля являются важными для описания самого факта продажи. Если хотя бы одно из них отсутствует, запись становится неполной или теряет смысл.

Для products атрибуты, которые не могут быть пустыми:
- product_id (уникальный идентификатор товара)
- product_name (товар без названия не идентифицируем для пользователя и системы)
- price (без цены товар нельзя использовать в продажах)
- category_id (товар должен принадлежать категории)
Эти атрибуты описывают основные характеристики продукта. Без них товар нельзя корректно использовать в системе (продать, классифицировать или отобразить).

2. Укажите, какие ID должны быть уникальными (UNIQUE).

Для sales уникальными должны быть атрибуты:
- sales_id
- transaction_number

Для products уникальными должны быть атрибуты:
- product_id

3. Предложите 2–3 ограничения CHECK (например, для числовых полей, которые не могут быть отрицательными, или для логических полей, которые могут принимать только два значения).

Для sales можно установить CHECK для атрибутов:
- quantity (только положительные значения)
- total_price (только положительные значения или нуль)
- discount (только в диапазоне от 0 до 100)

Для products можно установить CHECK для атрибутов:
- price (только положительные значения)
- resistant (только два значения)
- is_allergic (только два значения)
- vitality_days (только положительные значения или нуль)
